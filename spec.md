# Requirements

Let's write an aggregator (unicity oracle) implementation in Rust.


## Upstream

It certifies the root of SMT tree periodically by the BFT Core, an external service. There is an existing and working crate for it: ./crates/bft-committer .
- Request is state transfer: previous SMT root hash, next proposed SMT root hash, more metadata (see the crate), and the consistency proof.
- Request is signed using cryptographic key; validated by the BFT Core.
- BFT Core returns a certificate after some processing time - 100ms .. 1 second (it can be single machine or a cluster of machines)
- The next batch processing can start speculatively before BFT Core returns the certificate
- If there is error with BFT Core, previous SMT state must be rolled back. This is unlikely scenario but must be implemented
- SMT root hashes must be certified in a chain: for the next certification attempt, only previously certified root hash can be used as the previous root, etc.
- for reference and test environment setup, here is BFT Core implementation in go: ./bft-core

## Data

The SMT is in fact a path-compressed trie as defined by ./ndsmt3.py .
- We are optimizing for batch insert speed (number of insertions/sec)
- The typical usage pattern: 1. Collect client requests, 2. rite a batch, 3. get it certified by BFT core, 4. return inclusion proofs to clients who request them directly (ie, there is no callback mechanism, now). The inclusion proofs are required quite fast, in fw rounds. 5. Then, there can be much later very infrequent inclusion proof and non-inclusion proof queries (we are not optimizing to serve them fast).
- Note that ndsmt3.py is a prototype and we're not aiming format compatibility with it. It illustrates the right algorithmix logic.
- The tree is add-only. If a leaf exists its value is invariant. Leafs can not be changed or removed. Only new leafs can be added. Entire batch must not fail if it contains some pre-existing leaves. These duplicates may be ignored or logged if feasible wouthout performance penalty.
- We'll implement some features like consistency proofs later.

## Features in the roadmap (implemented after previous is complete)

### Consistency proofs

Consistency proofs as implemented in @ndsmt3.py : an operation log proving that no other leaves were modified or deleted, except insertions in the provided batch. The proof is presented to BFT Core in zkproof (or similar) field.

- We'll implement its verification at the BFT Core later.
- It is compile-time (if more efficient than just configuration file option) configuration option

### Scaling to disk

Scaling beyond the available memory by inserting leaves and necessary nodes to RocksDB k-v store, after completing processing of a batch. As an atomic transaction. Rolled back if certification by BFT Core does not succeed. Some caching must be implemented: the batch is already processed on in-memory data, but we need to cache the siblings (or whatever data needed) for some rounds to generate inclusion proofs for clients really fast.

- Keep leaves and other nodes in separate databases.
- Use RocksDB MultiGet at the beginning of round to fetch nessecary values for efficiency. Same applies to writes. If for single inclusion proof generation necessary sibling values are not cached then retrieve all as single MultiGet operation.
- As an optional extension, we use this feature as base of high availability implementation with one writer and many reader aggregator nodes.
- It is compile-time (if more efficient than just configuration file option) configuration option

- Utilizing all CPU cores by using e.g. Rayon library. This is complicated, the tree updates must be sharded to subtrees for it to make sense, and then results collected before root is reached. This would make consistency proof implementation more complicated. Number of threads is a configuration option. The performance benefit might be insignificant, especially if RocksDB is being used. Actual testing is needed.

- Good performance testing harness

- Make the client request/reply loop callback based for efficiency (so that there is no polling for inclusion proof); this also needs updates at client sdk level; we're focusing on batch insertion performance now.
- make the hashing algorithm configurable (sha256 is the default, we'd like to test other options' like blake3 performance).

# The required end-state:

0) There is no option without DB.
1) In-memory SMT. Snapshots must be implemented by copy-on-write mechanism for maximum efficiency. The state is persisted using RocksDB,
  with the goal of not losing state on crash or reset. This is the max. performance setup. Speculative execution is necessary - we must be
  doing work while waiting for UC from BFT Core.
2) The scale-to-disk option. Only necessary portion of leaves, internal nodes are kept in-memory. Most of the tree is on disk. Snapshots
  implemented using DB specific technology. Caching for 1) reasonable performance of batch insertion, and 2) fast responses to near immediate
  inclusion proof requests sent by clients who have just submitted (async) certification requests. This option is for scaling to maintain
  possibly very large SMT with large number of leaves, while the performance is degrading gracefully while the tree is growing (and is much
  larger than available memory).

The options 1) and 2) are mutually exclusive and can be chosen at compile time. Initially we ignore how to convert between them.
